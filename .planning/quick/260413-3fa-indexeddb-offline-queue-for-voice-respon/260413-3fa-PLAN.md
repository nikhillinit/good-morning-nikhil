<objective>
Implement an IndexedDB offline queue so that failed voice-response uploads are stashed
locally (sessionId, screenId, blob, mimeType) and retried on next app load or network
recovery, instead of blocking survey answer persistence.

Purpose: Voice uploads depend on Supabase Storage availability and network connectivity.
On mobile (the primary form factor), flaky connections should not prevent the user from
progressing through the survey. Stashing failed blobs in IndexedDB and retrying later
decouples answer persistence from upload success.

Output: `src/lib/voice-queue.ts` (new), updated `storage.ts` and
`screen-response-persistence.ts`, and comprehensive tests.
</objective>

  <action>
Create `src/lib/voice-queue.ts` with zero external dependencies — use the raw IndexedDB API directly (no `idb` library). The project has no IndexedDB dependency and adding one is unjustified for this simple key-value use case.

**Database schema:**
- DB name: `gmn-voice-queue`, version 1
- Object store: `pending-uploads`, keyPath: `key` (string = `${sessionId}/${screenId}`)
- Fields: `{ key, sessionId, screenId, blob: Blob, mimeType: string, createdAt: number }`

**Exported API:**

```typescript
export interface VoiceQueueEntry {
  key: string;
  sessionId: string;
  screenId: string;
  blob: Blob;
  mimeType: string;
  createdAt: number;
}

export async function enqueueFailedUpload(
  sessionId: string,
  screenId: string,
  blob: Blob,
  mimeType: string,
): Promise<void>;

export async function flushVoiceQueue(
  onUploaded?: (sessionId: string, screenId: string, publicUrl: string) => Promise<void>,
): Promise<{ flushed: number; failed: number }>;

export async function getQueueSize(): Promise<number>;
```

**Implementation details:**
- Wrap all IndexedDB access in a `getDb()` helper that returns `Promise<IDBDatabase>`. Cache the opened DB in a module-level variable. If `typeof indexedDB === "undefined"`, return null and all public functions short-circuit to safe defaults (enqueue is a no-op, getQueueSize returns 0, flushVoiceQueue returns `{flushed:0, failed:0}`).
- `enqueueFailedUpload` uses a readwrite transaction with `put` (not `add`) so re-enqueuing the same sessionId+screenId overwrites.
- `flushVoiceQueue` opens a readonly transaction, gets all entries via `getAll()`, then for each entry: calls `uploadVoiceResponse(entry.sessionId, entry.screenId, entry.blob)`. On success: delete the entry (new readwrite txn), call `onUploaded` if provided. On failure: increment failed counter, continue to next entry (do not throw).
- Import `uploadVoiceResponse` from `@/lib/supabase/storage` inside `flushVoiceQueue` to avoid circular dependency issues — use a dynamic import or top-level import is fine since storage.ts will not import voice-queue.ts.

**Test file `test/voice-queue.test.ts`:**

The test environment is jsdom which does not have IndexedDB. Use `fake-indexeddb` as a dev dependency (`npm install -D fake-indexeddb`). In the test file, import `fake-indexeddb/auto` at the top before importing the module under test. This globally polyfills `indexedDB`, `IDBDatabase`, etc.

**Step A — Update `src/lib/screen-response-persistence.ts`:**

Modify `resolvePersistableValue` to catch upload failures and enqueue instead of propagating the error. The function currently has:

```typescript
const mediaUrl = await uploadVoiceResponse(sessionId, screen.id, value.blob);
return { ...value, mediaUrl };
```

Replace with:

```typescript
import { enqueueFailedUpload } from "@/lib/voice-queue";

// Inside resolvePersistableValue, wrap the upload in try/catch:
try {
  const mediaUrl = await uploadVoiceResponse(sessionId, screen.id, value.blob);
  return { ...value, mediaUrl };
} catch (uploadError) {
  console.warn(
    `[voice-queue] Upload failed for ${screen.id}, queuing for retry`,
    uploadError,
  );
  await enqueueFailedUpload(
    sessionId,
    screen.id,
    value.blob,
    value.mimeType || value.blob.type || "audio/webm",
  );
  // Return the value WITHOUT mediaUrl so persistence proceeds.
  // response-contract.ts will produce an empty answers array for audio
  // without mediaUrl, which is acceptable — the answer gets patched
  // when flushVoiceQueue succeeds later.
  return { ...value, mediaUrl: undefined, blob: undefined };
}
```

This means `persistScreenResponse` no longer throws on upload failure. The survey progresses. The audio answer row will be empty until the queue flushes successfully.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client IndexedDB -> Supabase Storage | Blob data crosses from local storage to remote on retry |
| client -> Supabase Postgres | Answer patch on flush crosses to remote DB |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-q-01 | Tampering | voice-queue.ts IndexedDB store | accept | IndexedDB is same-origin sandboxed; attacker with local access already owns the browser. Low-value survey audio data. |
| T-q-02 | Denial of Service | enqueueFailedUpload | mitigate | Key is sessionId/screenId, so max entries = sessions x screens (bounded). No unbounded growth possible. |
| T-q-03 | Information Disclosure | IndexedDB blob storage | accept | Audio blobs are ephemeral survey responses, not PII. Cleared on successful flush. Same-origin policy applies. |
| T-q-04 | Spoofing | flushVoiceQueue onUploaded callback | mitigate | Callback receives sessionId/screenId from the queue entry and publicUrl from uploadVoiceResponse return value — both are validated by Supabase. No user-supplied URL injection. |
</threat_model>
